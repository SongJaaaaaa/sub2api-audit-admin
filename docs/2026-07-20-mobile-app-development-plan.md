# Sub2API 审计后台 App 开发计划

> 编制日期：2026-07-20  
> 适用项目：Sub2API 审计后台  
> 默认交付顺序：Android 优先，iOS 随后  
> 技术方案：Vue 3 + Vite + Capacitor，桌面 Web 与 App 共用业务代码

## 1. 项目目标

将当前管理后台交付为适合手机日常操作的 App，而不是简单把桌面网页放入 WebView。

本项目完成后应达到以下结果：

1. App 登录后进入模块首页，不显示桌面侧边栏或手机抽屉菜单。
2. 模块首页用两列宫格直接展示全部功能入口，不增加业务分类层级。
3. 手机端列表采用摘要卡片，完整字段进入详情页查看。
4. 查询、筛选、分页、详情、创建、充值、重试、确认等现有业务能力保持一致。
5. 桌面 Web 保留现有侧边栏和表格，不因 App 改造出现回归。
6. Android 可安装、升级和连接正式 HTTPS API；iOS 使用相同前端代码构建。
7. 关键流程具备自动化测试和真机验收记录。

## 2. 默认范围和边界

### 2.1 本期包含

- Capacitor 工程和 Android/iOS 平台配置。
- App 专用布局、模块首页和页面返回导航。
- 当前正式路由下全部业务页面的手机端适配。
- 手机端卡片列表、筛选面板、详情页、分页和操作栏。
- 登录令牌安全存储、网络异常、登录失效和 App 返回键处理。
- 附件上传、文件下载、日期选择、富文本和图表的 App 兼容。
- App 图标、启动屏、版本号、签名配置说明和发布清单。
- 桌面 Web、移动 Web、Android App 的自动化与人工验收。

### 2.2 本期不包含

- 重写为 Flutter、React Native 或原生 Android/iOS。
- 离线编辑、离线账本或离线提交财务操作。
- 推送通知、生物识别登录和设备审批，除非后续单独立项。
- 改变现有财务统计口径、权限模型或 Sub2API 数据边界。
- 未注册到正式路由的页面，例如当前未启用的 `GiftQuotaListView.vue`。

### 2.3 默认产品决策

- App 首页路径使用独立路由，避免改变桌面 Web 的现有首页。
- App 内不设置侧边栏、抽屉菜单或常驻底部模块菜单。
- 模块内通过顶部返回按钮回到模块首页；Android 同时支持系统返回键。
- 列表卡片只显示 3 至 5 个关键字段，其他字段进入详情页。
- App 不缓存敏感业务数据供离线查看；无网络时只显示明确错误和重试入口。
- Android 首次交付内部测试包；商店或企业分发作为发布阶段处理。

## 3. 当前基础

### 3.1 可直接复用

- Vue 3、TypeScript、Vue Router、Pinia 和现有组件库。
- 现有登录、权限守卫、查询参数、分页状态和 API 类型。
- 11 个正式业务入口及其后端接口。
- 桌面表格、统计卡片、详情抽屉和表单业务逻辑。
- Laravel Bearer Token 鉴权及服务端 Token 注销。
- Playwright 桌面和 iPhone 12 两套测试项目。
- Laravel Feature 测试、生产部署流程和 HTTPS Web API。

### 3.2 必须调整

- 当前手机导航使用抽屉，需要增加 App 专用布局。
- 当前多数列表依赖宽表格和横向滚动，需要增加卡片列表。
- 当前 Token 存储在 `localStorage`，App 应迁移至安全存储。
- 当前 API 默认使用相对地址 `/api/v1`，App 构建必须使用正式绝对 HTTPS 地址。
- 文件下载依赖浏览器链接点击，需要适配原生文件保存与分享。
- 需要配置 App WebView 的精确 CORS 来源，禁止使用通配来源。

## 4. App 信息架构

### 4.1 页面结构

```text
登录
  -> App 模块首页
      -> 数据总览
      -> 消费排行
      -> Sub2API 用户
      -> 用户充值
      -> 收入
      -> 支出
      -> 历史账
      -> 利润统计
      -> 异常中心
      -> 操作审计
      -> 管理员账号
```

### 4.2 模块首页

模块首页使用单层两列宫格，所有模块直接平铺，不显示“数据、用户、财务、管理”等分类标题，也不增加分类中间页。模块项包含：

- 图标。
- 模块名称。
- 一行简短状态或关键数字；仅展示现有接口能够可靠提供的数据。
- 明确的点击状态和足够大的触控区域。

首页顶部显示 App 名称、当前管理员和退出入口。主题切换可放入右上角菜单，不占用模块区域。模块顺序按照实际使用频率排列，点击模块项直接进入对应业务页面。

### 4.3 模块页面导航

- 一级模块页顶部固定返回按钮、模块标题和主要操作。
- 列表进入详情后，返回列表时保留筛选条件、滚动位置和已加载页数。
- 新增、充值、确认分账等写操作离开前，如有未提交内容需二次确认。
- Android 系统返回键顺序为：关闭弹层、返回详情来源、返回模块首页、二次返回退出 App。

## 5. 手机端列表设计规范

### 5.1 通用结构

```text
顶部标题与主要操作
搜索框 + 筛选按钮
已启用筛选条件
关键汇总
摘要卡片列表
加载更多 / 没有更多
```

### 5.2 卡片规则

- 第一行：记录主体和状态。
- 第二行：最重要的金额或业务指标。
- 第三行：时间、操作人或来源。
- 右侧使用熟悉的进入详情图标，不堆放多个文字按钮。
- 危险操作不直接放在卡片表面，进入详情后执行。
- 卡片高度应稳定，长邮箱和长备注最多显示两行并省略。

### 5.3 筛选规则

- 简单关键词保留在列表顶部。
- 其他条件统一放到底部筛选面板。
- 面板底部固定“重置”和“查询”。
- 列表顶部显示当前生效的筛选标签和结果数量。
- 日期范围使用适合触控的选择方式，不直接照搬桌面宽控件。

### 5.4 分页规则

- App 默认每次加载 20 条。
- 用户主动点击“加载更多”，避免误触和无止境请求。
- 新查询、筛选变化或刷新时重置到第一页。
- 加载失败只重试失败页，不清空已经显示的数据。
- 桌面端继续使用页码分页。

### 5.5 详情与操作规则

- 简单只读详情使用全屏详情页或接近全屏的底部面板。
- 复杂表单使用独立页面，避免键盘遮挡弹窗。
- 提交按钮固定在安全区域上方。
- 财务写操作必须显示对象、金额、操作类型和影响结果的确认摘要。
- 请求提交期间禁用重复提交，成功后刷新来源列表和汇总。

## 7. 技术设计

### 7.1 工程结构

计划新增以下薄层组件，业务请求仍留在现有页面中：

```text
frontend/src/
  app/
    platform.ts
    storage.ts
  layouts/
    AppLayout.vue
  views/
    AppHomeView.vue
  components/mobile/
    AppPageHeader.vue
    MobileFilterSheet.vue
    MobileListState.vue
    MobileLoadMore.vue
    MobileActionBar.vue
  styles/
    app.css
  composables/
    useAppMode.ts
    useMobileList.ts
```

只抽取多个页面确实重复的导航、筛选和加载状态。每种业务卡片直接写在对应页面，避免为了统一而制造难读的动态字段配置。

### 7.2 App 模式识别

- 原生包内通过 Capacitor 平台能力识别 App。
- 浏览器开发和 E2E 可通过 `VITE_APP_MODE=app` 模拟 App 布局。
- 不仅依赖屏幕宽度判断，防止普通手机浏览器被误认为原生 App。
- 屏幕宽度只用于卡片布局、间距和控件尺寸。

### 7.3 路由

- 新增 App 模块首页路由，例如 `/app`。
- App 登录成功跳转 `/app`；桌面登录成功仍进入 `/`。
- 现有业务路由保持不变，避免影响接口、书签和已有测试。
- 路由元数据增加标题、App 是否显示、首页顺序和返回目标。
- AppLayout 统一读取路由元数据渲染标题和返回按钮。

### 7.4 API 与环境变量

- Web 生产环境继续使用同域 `/api/v1`。
- App API 地址固定为 `https://audit.sjiaa.cc.cd/api/v1`，构建无需设置 `VITE_API_BASE_URL`。
- 正式环境只允许 HTTPS，不接受明文 HTTP API。
- 后端 CORS 仅加入实际 Capacitor 来源和必要请求头。
- 验证 `Authorization`、文件上传、文件下载和 401 响应均能通过 WebView。
- 不在 App 包、日志或构建产物中写入服务端 API Key、数据库密码等秘密。

### 7.5 登录与安全存储

- 保持现有 Bearer Token 接口，不修改服务端鉴权协议。
- 增加统一的 Token Storage 接口：Web 使用 `localStorage`，App 使用原生安全存储。
- 首次升级时读取旧存储并迁移，成功后删除 App WebView 中的旧 Token。
- App 冷启动读取安全存储，再调用 `/auth/me` 验证 Token。
- 401 时清理安全存储并跳转登录页。
- 主动退出调用服务端注销接口，然后清理本地 Token。
- 生产日志不得记录 Token、Authorization 或完整敏感响应。

### 7.6 原生能力

- 安全区域：兼容刘海、状态栏和底部手势区。
- 状态栏：跟随当前明暗主题，确保图标可见。
- 键盘：输入时页面可滚动，底部提交按钮不被遮挡。
- 返回键：按页面层级关闭弹层或返回上一页。
- 网络状态：断网时显示明确状态，恢复网络后允许手动重试。
- 文件上传：支持文件选择；相机和相册入口按实际附件需求决定。
- 文件下载：使用原生文件系统保存并调用系统分享或打开能力。
- 外部链接：交给系统浏览器打开，不在管理 WebView 中任意跳转。

### 7.7 桌面与 App 共存

- 数据加载、筛选和提交函数只保留一份。
- 桌面使用现有 `a-table`，App 使用移动卡片模板。
- 页面按 App 模式切换展示，不复制整套页面文件。
- 现有桌面列设置只在桌面显示。
- 桌面详情抽屉可保留；App 详情改为全屏布局。

## 8. 分阶段实施计划

### 阶段 0：范围确认与界面基线，1 至 2 人日

任务：

- 确认 App 名称、图标、包名、支持系统版本和分发方式。
- 为模块首页、卡片列表、筛选面板、详情页和写操作确认稿。
- 确认每个列表卡片必须显示的字段。
- 确认 Android 首批测试设备和 iOS 构建账号。

交付：

- 页面清单和字段矩阵。
- 关键页面移动端原型。
- App 标识信息清单。

### 阶段 1：Capacitor 与 App 基础设施，2 至 3 人日

任务：

- 安装并初始化 Capacitor。
- 添加 Android 工程；在可用的 macOS 环境添加 iOS 工程。
- 配置开发、测试和生产 API 地址。
- 建立 App 模式识别与统一存储接口。
- 接入安全区域、状态栏、键盘、网络和返回键。
- 验证登录、Token 恢复、401 和退出。

验收：

- Android Debug 包可安装并启动。
- App 可登录并访问正式格式的 HTTPS API。
- 冷启动保持登录，Token 失效后回到登录页。

### 阶段 2：App 布局与模块首页，2 至 3 人日

任务：

验收：

## 9. 测试计划

### 9.1 自动化测试

保留现有测试，并增加 App 模式覆盖：

- 桌面 Chrome：验证现有侧边栏、表格和业务流程。
- iPhone 12 尺寸：验证移动布局。
- Pixel 7 尺寸：验证 Android 常见视口。
- App 模式：验证模块首页、无抽屉导航、返回流程和卡片列表。

首批 E2E 场景：

1. 登录后进入模块首页。
2. 模块入口可以打开所有正式业务页面。
3. 用户查询、筛选、加载更多和详情返回。
4. 用户充值成功、失败和防重复提交。
5. 异常记录查看、重试和作废确认。
6. 收入、支出新增及列表刷新。
7. 历史账筛选和导出。
8. 利润确认和撤销。
9. Token 失效后回到登录页。
10. 桌面端原有流程无回归。

### 9.2 视觉验收尺寸

- 320 x 568：极窄 Android 基线。
- 375 x 812：常见 iPhone。
- 390 x 844：现代 Android/iPhone。
- 430 x 932：大屏手机。
- 768 x 1024：平板。
- 1440 x 900：桌面回归。

每个尺寸检查：

- 页面无非预期横向滚动。
- 标题、金额、状态和按钮不重叠。
- 长邮箱、长模型名和长备注不会撑破卡片。
- 键盘弹出后当前输入和提交按钮仍可操作。
- 深色和浅色模式均有足够对比度。

### 9.3 真机验收

- 首次安装、覆盖升级和卸载重装。
- Wi-Fi、移动网络、断网和网络恢复。
- App 切到后台后恢复。
- Android 系统返回键和 iOS 边缘返回。
- 文件选择、上传、下载、打开和分享。
- HTTPS 证书、登录保持和 Token 失效。
- 不同系统字体大小下无关键内容遮挡。

## 10. 验收标准

### 10.1 导航

- App 不出现桌面侧边栏、手机抽屉或汉堡菜单。
- 登录后首页能直接看到所有模块，不出现分类层级或分类中间页。
- 任一模块最多一次点击进入。
- 任一一级模块都能明确返回模块首页。

### 10.2 列表

- 手机端不依赖横向表格滚动完成主要工作。
- 卡片展示模块最重要的 3 至 5 个字段。
- 全部桌面字段可在详情中查看。
- 查询、筛选、排序和分页结果与桌面端一致。
- 空数据、加载、失败和没有更多数据均有明确状态。

### 10.3 写操作

- 所有金额操作提交前显示确认摘要。
- 提交时禁止重复点击。
- 成功后相关列表、详情、余额和统计同步刷新。
- 部分成功的批量操作能分别展示成功和失败结果。
- 401、403、409、422 和 502 能显示现有业务错误，不显示假成功。

### 10.4 性能

- 冷启动后首个可操作页面目标不超过 3 秒，具体以测试网络和设备基线记录。
- 模块首页只请求必要数据，不为每个模块单独发起重复请求。
- 长列表分页加载时不阻塞点击和滚动。
- 图表页面离开后释放监听和图表实例。

### 10.5 安全

- App Token 不长期保存在 WebView `localStorage`。
- 正式 API 全部使用 HTTPS。
- CORS 不使用 `*` 放开管理接口。
- App 包中不存在数据库密码、Sub2API API Key 或签名密码。
- 日志不记录密码、Token、Authorization 和敏感完整响应。

## 12. 风险与处理

| 风险 | 影响 | 处理方式 |
|---|---|---|
| 直接复用宽表格 | 手机难以浏览和操作 | 卡片摘要 + 全屏详情，桌面保留表格 |
| App 使用相对 API 地址 | 原生包请求到错误来源 | App 构建强制绝对 HTTPS API 地址 |
| Token 保存在 localStorage | 设备环境中保护不足 | 统一存储接口并迁移到安全存储 |
| CORS 配置不完整 | App 登录或上传失败 | 只配置实际 WebView 来源并做真机验证 |
| Blob 下载在 WebView 失效 | 历史账导出无法保存 | 使用原生文件系统和分享能力 |
| 键盘遮挡复杂表单 | 充值或支出无法提交 | 独立表单页、可滚动内容、底部安全区 |
| 图表固定尺寸 | 小屏裁切或空白 | 容器约束、进入页面重绘、真机截图检查 |
| iOS 无构建环境 | 无法生成 Archive | 提前准备 macOS、Xcode、证书和账号 |
| 桌面共用代码回归 | 现有后台受影响 | 双模式 E2E 和桌面发布门禁 |
| 首页模块请求过多 | 启动慢、接口压力增加 | 聚合现有总览数据，不逐模块重复请求 |

## 13. 发布与回滚

### 13.1 Android

- Debug APK 用于开发和接口联调。
- Release APK 用于受控内部测试。
- AAB 用于 Google Play 或支持 AAB 的分发渠道。
- 签名文件、别名和密码存放在仓库外，提供单独保管说明。
- 每次发布递增 `versionCode`，用户可通过覆盖安装升级。

### 13.2 iOS

- 使用独立 Bundle ID 和正式签名。
- 先发布 TestFlight 内测，再决定 App Store 或企业分发。
- iOS 打包必须在 macOS + Xcode 环境完成。

### 13.3 回滚

- App 发布前保留上一版安装包和版本号。
- 后端接口保持向后兼容，避免 App 更新与服务端发布必须同时发生。
- App 出现严重问题时停止新版本分发，并恢复上一稳定版本。
- 本次原则上不修改数据库结构；如后续新增首页聚合接口，迁移和回滚另行评审。

## 15. 开发执行顺序

实际编码按以下顺序推进，不并行大范围改造页面：

1. 确认 App 标识和关键页面字段。
2. 接入 Capacitor、App 模式和安全存储。
3. 完成 AppLayout、模块首页和返回逻辑。
4. 用 Sub2API 用户与异常中心验证通用列表方案。
5. 完成用户充值并验证复杂写操作。
6. 批量完成四个财务页面。
7. 完成数据总览、排行、审计和管理员页面。
8. 补齐原生上传下载、键盘、网络和返回键。
9. 执行双模式自动化、真机验收和桌面回归。
10. 生成签名包，先 Android 内测，再推进 iOS。

每完成一个阶段立即验收并修复，不把所有问题集中到发布前处理。
