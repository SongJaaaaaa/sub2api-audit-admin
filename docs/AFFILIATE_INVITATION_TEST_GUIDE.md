# 邀请与一级返利测试指南

## 1. 测试目标

本指南用于验证以下完整链路：

```text
A 获取邀请链接
-> B 通过链接注册 Sub2API
-> B 登录本系统，同步 A -> B 关系
-> B 产生有效充值/兑换事件
-> 扫描器入库，队列计算返利
-> A 获得一级返利
```

核心规则：

- 只奖励直接上级。A 邀请 B、B 邀请 C 时，B 充值只奖励 A，C 充值只奖励 B。
- 没有有效直接上级时不向更上级补发。
- 同一来源事件重复扫描、重复执行队列，不得重复返利。
- 金额以管理端“返利配置”当前值为准，不要只按本文示例判断。

## 2. 当前环境前置检查

### 2.1 配置邀请链接模板

截至 2026-07-16，推广接口能够返回邀请码，但本地 `invite_url` 为空。测试前在 `backend\.env` 配置：

```env
REBATE_INVITE_URL_TEMPLATE=https://api.sjiaa.cc.cd/register?aff={code}
```

线上 Sub2API 已确认支持 `/register?aff=<邀请码>`。`aff` 是推广归因参数，不是页面上可见的“优惠码”。打开链接后优惠码输入框为空属于正常现象，不要把邀请码手工填入优惠码。

修改后重启 Laravel 服务；如果生产环境启用了配置缓存，再执行：

```powershell
Set-Location 'D:\test项目\sub2apiAccount\sub2api-audit-admin\backend'
& 'D:\php\php.exe' artisan config:clear
```

重新登录普通用户并打开“推广中心”，复制出的地址应类似：

```text
https://api.sjiaa.cc.cd/register?aff=V6JEWBPERNTL
```

不要在测试文档或截图中记录密码、Token、Admin API Key。

### 2.2 准备测试账号

准备三个 Sub2API 普通用户：

| 代号 | 用途 | 要求 |
|---|---|---|
| A | 第一位邀请人 | 已有普通账号，可以登录本系统 |
| B | A 的直接下级 | 必须使用未注册过的新邮箱，通过 A 的链接注册 |
| C | B 的直接下级 | 完整一级隔离测试时使用，也必须是新邮箱 |

已有账号不能通过事后打开邀请链接补建关系。不要使用已有生产账号充当 B 或 C，也不要直接修改本地返利关系表。

### 2.3 检查返利配置和切换时间

管理员从统一登录页进入系统，打开：

```text
http://127.0.0.1:5173/rebate/config
```

记录以下配置：

| 配置 | 测试建议 |
|---|---|
| 原生充值参与返利 | 使用原生充值测试时开启 |
| 兑换参与返利 | 使用兑换码测试时开启 |
| 后台调额参与返利 | 保持关闭 |
| 返利切换时间 | 必须早于本次测试事件时间 |

`rebate_cutover_at` 是一次性切换点。若页面已经显示切换时间，不要重置，也不要再次执行切换命令。只有首次部署、字段为空且已确认要开始接收新事件时，才执行一次：

```powershell
Set-Location 'D:\test项目\sub2apiAccount\sub2api-audit-admin\backend'
& 'D:\php\php.exe' artisan rebate:cutover
```

## 3. 最短冒烟测试

### 步骤 1：A 获取邀请链接

1. 在 `http://127.0.0.1:5173/login` 使用 A 的 Sub2API 账号登录。
2. 普通用户应自动进入推广客户端，不需要选择“管理员/推广员”。
3. 打开“推广中心”。
4. 记录 A 的邀请码、直接邀请人数和当前可用返利余额。
5. 复制邀请链接，确认它包含 `https://api.sjiaa.cc.cd/register?aff=` 和 A 的邀请码。

预期结果：邀请链接不为空，复制按钮可用。

### 步骤 2：B 通过 A 的链接注册

1. 使用无痕窗口打开 A 的完整邀请链接。
2. 使用新的测试邮箱注册 B。
3. 完成 Sub2API 要求的邮箱验证或验证码。
4. 保存注册时间和 B 的 Sub2API 用户 ID，不记录密码。

预期结果：B 注册成功，注册链接中的 `aff` 参数没有在跳转过程中丢失。

### 步骤 3：B 登录本系统并同步关系

1. 打开 `http://127.0.0.1:5173/login`。
2. 使用 B 的 Sub2API 邮箱和密码登录。
3. 确认 B 自动进入 `/affiliate/dashboard`。
4. 退出 B，重新登录 A，查看“我的团队”。

预期结果：A 的直接邀请人数增加 1，“我的团队”中出现 B。

管理员也应在 `/rebate/relationships` 搜索 B，确认：

```text
B.parent_user_id = A.id
B.parent_email = A.email
```

推荐关系以 Sub2API 的 `user_affiliates.inviter_id` 为事实来源。B 登录本系统时会同步本地关系；若 B 未登录，也会在 B 的新充值事件被扫描时同步。

### 步骤 4：B 产生一笔有效测试事件

优先使用专门的测试兑换码；也可以使用测试环境的原生充值订单。事件必须发生在返利切换时间之后。

不要使用以下方式做默认测试：

- 后台手工调额：默认关闭，不会参与返利。
- 纯赠送调额：固定排除。
- 返利提现产生的调额：固定排除，避免形成返利回声。
- 真实生产资金：测试应使用隔离账号和可核销的测试金额。

若当前配置为：

```text
初始累充门槛       100.00
初始每次奖励        15.00
初始最多奖励次数     2
后续累充门槛       100.00
后续每次奖励        15.00
```

则让从未产生过返利进度的 B 新增一笔 `100.00` 的有效事件，A 应获得 `15.00`。

### 步骤 5：运行扫描和队列

Scheduler 正常运行时可以等待自动处理。为了让测试时间可控，可按事件来源手工执行。

兑换事件：

```powershell
Set-Location 'D:\test项目\sub2apiAccount\sub2api-audit-admin\backend'
& 'D:\php\php.exe' artisan rebate:scan --source=redeem --limit=200
& 'D:\php\php.exe' artisan rebate:recover-queue
& 'D:\php\php.exe' artisan queue:work --stop-when-empty --tries=3
```

原生充值事件：

```powershell
Set-Location 'D:\test项目\sub2apiAccount\sub2api-audit-admin\backend'
& 'D:\php\php.exe' artisan rebate:scan --source=native_recharge --limit=200
& 'D:\php\php.exe' artisan rebate:recover-queue
& 'D:\php\php.exe' artisan queue:work --stop-when-empty --tries=3
```

不要为处理单个测试事件重新执行 `rebate:cutover`。

### 步骤 6：核对结果

登录 A，核对“仪表盘”和“返利明细”：

| 核对项 | 预期结果 |
|---|---|
| 可用返利余额 | 在原余额基础上增加 `15.00` |
| 付款用户 | B |
| 返利层级 | `1` |
| 来源金额 | `100.00` |
| 返利金额 | `15.00` |
| 记录状态 | 已完成/成功 |

再登录 B，确认这笔充值不会奖励 B 本人。

## 4. 完整 A-B-C 一级隔离测试

1. B 登录本系统，在“推广中心”复制 B 的邀请链接。
2. 使用新邮箱 C 通过 B 的链接注册。
3. C 登录本系统，使 B -> C 关系同步。
4. 管理端搜索 C，确认直接上级是 B，不是 A。
5. 记录 A、B 的返利余额。
6. 让 C 产生一笔切换时间后的 `100.00` 有效事件。
7. 运行对应来源扫描和队列。

预期结果：

| 用户 | 余额变化 | 团队可见范围 |
|---|---:|---|
| A | `0.00` | 只显示 B，不显示 C |
| B | `+15.00` | 显示 C |
| C | `0.00` | 不因本人充值获得邀请返利 |

如果 A 因 C 的充值获得返利，说明出现了多级返利，测试失败。

## 5. 幂等和多门槛测试

### 5.1 重复扫描

完成一笔返利后，原样再次运行相同来源的扫描、恢复和队列命令。

预期结果：

- A 的余额不再增加。
- 返利记录数量不增加。
- 同一来源业务键只对应一个返利事件。

### 5.2 单笔跨多个门槛

使用全新下级账号，确保累计进度为 `0.00`。在上述示例配置下产生一笔 `450.00` 有效事件。

预期计算：

```text
2 次初始里程碑 x 15.00 = 30.00
2 次后续台阶   x 15.00 = 30.00
直接上级合计             = 60.00
剩余未达门槛进度         = 50.00
```

预期结果：直接上级一次处理获得 `60.00`，重复扫描后金额不变。

### 5.3 无上级账号

使用未携带任何 `aff` 参数注册的全新账号，登录本系统后产生有效事件。

预期结果：不产生返利记录，不向其他祖先或任意用户补发。

## 6. 问题排查

遇到失败时先保留请求时间、账号 ID、来源类型和来源业务 ID，再查看日志，不要直接修改关系或补余额。

| 现象 | 首先检查 |
|---|---|
| 推广中心邀请码存在，但邀请链接为空 | `REBATE_INVITE_URL_TEMPLATE` 是否配置，Laravel 是否已重启/清除配置缓存 |
| 注册页优惠码为空 | 正常；邀请归因来自 URL 的 `aff` 参数，不是优惠码 |
| B 不在 A 的团队中 | B 是否通过完整链接注册；B 是否登录过本系统；Sub2API `user_affiliates.inviter_id` 是否为 A |
| 管理端搜索用户失败 | Laravel 服务、Sub2API 隧道/Admin API 状态及对应请求日志 |
| 扫描结果为 0 | 事件是否晚于切换时间，来源开关是否开启，执行的 `--source` 是否正确 |
| 有事件但没有返利 | 队列是否运行，事件状态和失败原因，B 是否存在有效直接上级 |
| 金额与预期不同 | 测试前实际配置、该下级已有累计进度、事件金额和来源是否正确 |
| 出现重复返利 | 两条记录的来源类型、来源 ID、事件 ID和幂等键 |

Laravel 日志：

```text
D:\test项目\sub2apiAccount\sub2api-audit-admin\backend\storage\logs\laravel.log
```

只截取测试发生时间附近的日志。若需要协助定位，请提供脱敏后的相关日志、浏览器请求响应和测试记录，不要提供密码、Token 或密钥。

## 8. 通过标准

- 新用户通过邀请链接注册后，直接上级关系正确且可在管理端查询。
- A-B-C 只按直接上级返利，不越级、不补发。
- 单笔跨门槛金额正确，重复扫描和重复队列不重复入账。
- 事件发生在切换时间前、来源关闭或无有效上级时不返利。
- 测试全过程不修改历史资金流水，不泄露账号密码、Token 和密钥。
