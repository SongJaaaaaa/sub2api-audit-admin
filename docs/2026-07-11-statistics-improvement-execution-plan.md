# 后台统计完善执行计划

> 编制日期：2026-07-11
> 适用项目：Sub2API 审计后台
> 时区：Asia/Shanghai

## 1. 目标

本计划用于完善后台各业务表格的汇总统计、查询筛选和风险概览，重点解决以下问题：

1. 财务页面只有明细，没有完整金额汇总。
2. 调增和调减金额口径混合，容易产生误解。
3. 异常页面缺少风险金额、影响人数和未处理时长。
4. 部分统计可能只针对当前分页，不能代表完整查询结果。
5. 不同页面的统计卡片、接口结构和移动端样式不统一。

本次统计遵循以下原则：

- 所有统计基于完整查询条件，不基于当前分页数据计算。
- 列表和汇总使用完全相同的筛选条件。
- 本地账本与 Sub2API 官方统计明确区分数据来源。
- Sub2API 官方接口不可用时显示“暂不可用”，不使用 `0` 代替真实数据。
- 金额明确区分调增、调减、绝对金额和净金额。
- 用户数统一按用户 ID 去重。
- 图表只用于趋势、排行和分类分布，不为简单数字增加无意义图表。
- 所有新增统计兼容 H5，桌面端和移动端使用相同业务口径。

# 3. P0 实施内容

## 3.1 额度调整记录

### 当前问题

当前页面只有：

- 记录数
- 用户数
- 调整金额合计

“调整金额合计”会把调增和调减都按正数累加。例如调增 100、调减 80 时显示 180，不能表达真实净变化 20。

### 改造内容

将顶部统计调整为：

| 字段 | 口径 |
|---|---|
| 记录数 | 当前查询条件下的全部记录数 |
| 用户数 | 当前查询条件下去重用户数 |
| 调增金额 | `operation=increment` 的 `amount` 合计 |
| 调减金额 | `operation=decrement` 的 `amount` 合计 |
| 净调整金额 | 调增金额减去调减金额 |
| 现金入账 | `cash_amount` 合计 |
| 赠送额度 | `gift_quota_amount` 合计 |

### 后端接口

扩展 `GET /api/v1/ledger-adjustments` 返回：

```json
{
  "items": [],
  "total": 0,
  "page": 1,
  "page_size": 20,
  "summary": {
    "record_count": 0,
    "user_count": 0,
    "increment_total": "0.00",
    "decrement_total": "0.00",
    "net_total": "0.00",
    "cash_total": "0.00",
    "gift_total": "0.00"
  }
}
```

### 前端要求

- 顶部统计卡片随邮箱、操作人和时间段同步刷新。
- 净调整金额大于 0 显示绿色，小于 0 显示红色，等于 0 使用普通颜色。
- H5 使用两列布局，极窄屏使用一列。
- 删除含义不明确的“调整金额合计”。

### 验收标准

- 调增 100、调减 80 时，调增显示 100、调减显示 80、净调整显示 20。
- 分页切换不改变统计结果。
- 查询条件改变后，列表和汇总同步变化。

## 3.2 现金账与赠送额度

### 当前问题

当前页面只有两个明细表和用户 ID 筛选，没有金额、用户和关联情况汇总。

### 筛选改造

两个 Tab 统一增加：

- 用户 ID
- 用户邮箱
- 时间段
- 操作人
- 业务单号
- 关联状态：全部、已关联、未关联

切换现金账和赠送额度 Tab 时保留公共筛选条件。

### 现金账统计

| 字段 | 口径 |
|---|---|
| 现金入账合计 | 当前查询结果现金金额合计 |
| 记录数 | 当前查询结果记录数 |
| 用户数 | 去重用户数 |
| 已关联数 | 已关联本地额度调整的记录数 |
| 未关联数 | 没有本地额度调整关联的记录数 |

### 赠送额度统计

| 字段 | 口径 |
|---|---|
| 赠送额度合计 | 当前查询结果赠送金额合计 |
| 记录数 | 当前查询结果记录数 |
| 用户数 | 去重用户数 |
| 关联现金账数 | 能找到同业务调整现金账的记录数 |
| 无现金关联数 | 找不到对应现金账的记录数 |

### 后端接口建议

现金账和赠送额度接口统一返回：

```json
{
  "items": [],
  "total": 0,
  "page": 1,
  "page_size": 20,
  "summary": {
    "record_count": 0,
    "user_count": 0,
    "amount_total": "0.00",
    "linked_count": 0,
    "unlinked_count": 0
  }
}
```

赠送额度可额外返回：

```json
{
  "related_cash_count": 0,
  "missing_cash_count": 0
}
```

### 验收标准

- 汇总统计基于全部查询结果，而不是当前页。
- 点击赠送额度详情仍能查看关联现金账。
- 关联状态筛选与顶部关联数量一致。
- 无法关联的数据明确显示“未找到关联现金账”。

## 3.3 经营账

### 当前问题

页面支持分类和时间筛选，但没有支出合计、分类分布和支出规模概览。

### 新增统计

| 字段 | 口径 |
|---|---|
| 支出合计 | 当前查询条件下金额合计 |
| 支出笔数 | 当前查询结果记录数 |
| 分类数 | 去重分类数量 |
| 最大单笔支出 | 当前查询条件下最大金额 |
| 日均支出 | 支出合计除以查询日期天数 |

未选择时间范围时，不展示日均支出，避免使用不明确的时间范围计算。

### 新增筛选

- 操作人
- 金额最小值
- 金额最大值
- 备注关键词

### 分类图表

增加 ECharts 横向条形图：

- Y 轴：支出分类。
- X 轴：支出金额。
- 按金额从高到低排序。
- Tooltip：分类、金额、笔数、金额占比。
- 分类较多时使用 `dataZoom`。
- 图表颜色使用橙红色系，与收入、消费榜颜色区分。

### 后端返回建议

```json
{
  "summary": {
    "record_count": 0,
    "category_count": 0,
    "amount_total": "0.00",
    "max_amount": "0.00",
    "daily_average": null
  },
  "categories": [
    {
      "category": "服务器",
      "record_count": 0,
      "amount_total": "0.00"
    }
  ]
}
```

### 验收标准

- 新增、修改或删除经营支出后，列表、汇总和分类图同步刷新。
- 分类筛选后，图表只展示符合条件的分类结果。
- 日均支出的查询天数包含开始日和结束日。

## 3.4 异常中心

### 当前问题

当前页面主要展示异常记录，缺少异常规模、金额影响和未处理时长统计。

### 新增筛选

- 用户 ID
- 用户邮箱
- 异常类型
- 操作人
- 时间段
- 状态：待处理、已处理
- 金额范围

如果当前数据表没有“已处理”字段，本阶段先增加异常类型和时间段筛选；处理状态需要单独增加数据库字段和处理操作。

### 新增统计

| 字段 | 口径 |
|---|---|
| 异常总数 | 当前查询条件下异常记录数 |
| 用户数 | 去重用户数 |
| 涉及金额 | 异常记录 `amount` 合计 |
| 待处理数 | 尚未标记处理的异常数 |
| 超过 24 小时 | 创建超过 24 小时仍未处理的数量 |
| 最早异常时间 | 当前待处理异常中最早创建时间 |

### 异常类型分布

增加 ECharts 横向条形图，按异常数量从高到低展示：

- Sub2API 调用失败
- 二次确认余额不一致
- 作废记录
- 远端关联失败
- 数据字段异常
- 其他异常

Tooltip 展示异常类型、数量、涉及用户数和涉及金额。

### 数据结构扩展建议

如需真正管理异常处理状态，增加：

- `resolved_at`
- `resolved_by`
- `resolution_notes`

本阶段不要仅根据备注文本推断是否处理。

### 验收标准

- 异常数量、金额和列表使用相同查询条件。
- 超时异常按 Asia/Shanghai 计算。
- 没有处理状态字段时不显示虚假的“待处理数”。

## 4.2 对账中心

### 新增筛选

- 业务日期范围
- 对账状态
- 是否存在外部调额
- 是否存在审计孤儿
- 操作人

### 新增统计

- 对账批次数
- 正常批次数
- 告警批次数
- 错误批次数
- 差异记录数
- 差异金额
- 对账正常率
- 最近成功对账日
- 未对账天数

### 口径

```text
对账正常率 = 正常批次数 ÷ 总批次数
```

`warning` 不计入正常批次，单独展示告警数量。

### 验收标准

- 重跑对账后，批次列表、汇总和详情同步刷新。
- 差异金额只统计具有金额语义的差异，不给用户、方向等非金额差异强行填 0。
- 未对账天数按应对账日期计算，不包含未来日期。

## 4.3 模型消耗统计

### 新增顶部汇总

- 模型数
- 请求总数
- Token 总数
- 缓存 Token
- 缓存占比
- 标准消费
- 实际消费
- Top 3 模型 Token 占比

### 口径

- 数据完全来自 Sub2API 官方模型统计。
- 模型语义固定为 `requested`。
- 汇总由官方返回模型列表累加。
- 模型数据不可用时，整个统计区域显示不可用，不显示零值。

### Top 3 占比

```text
Top 3 模型 Token 占比 = Token 排名前三模型总 Token ÷ 全部模型总 Token
```

### 验收标准

- 模型筛选和日期变化后，汇总同步刷新。
- 模型榜总 Token 与顶部 Token 总数一致。
- Tooltip 和汇总字段均使用官方数据。

# 6. 统一后端实现规范

## 6.1 查询结构

每个列表接口采用同一个基础 Query：

1. 创建基础查询。
2. 应用所有筛选条件。
3. 克隆查询计算 `summary`。
4. 原查询进行排序、分页并返回 `items`。

不要在前端遍历当前页计算统计。

## 6.2 金额格式

- 后端金额统一返回字符串。
- 金额保留两位小数。
- 不使用 JavaScript 浮点数累加财务金额。
- 净金额允许返回负数。

## 6.3 时间范围

- 页面使用自然日期 `YYYY-MM-DD`。
- 时区固定为 `Asia/Shanghai`。
- 后端统一转换为左闭右开范围：

```text
[start_date 00:00:00, end_date + 1 day 00:00:00)
```

## 6.4 汇总返回结构

列表接口统一使用：

```json
{
  "items": [],
  "total": 0,
  "page": 1,
  "page_size": 20,
  "summary": {}
}
```

图表需要分类数据时增加：

```json
{
  "breakdown": []
}
```

## 6.5 统计错误处理

- 本地数据库汇总失败：返回明确接口错误。
- Sub2API 官方统计失败：返回统一 `SUB2API_STATS_UNAVAILABLE`。
- 不允许 catch 后返回空数组和零统计伪装成功。
- 如字段不清楚，先记录日志确认真实数据，不增加复杂猜测处理。

# 7. 统一前端实现规范

## 7.1 统计卡片

- 桌面端使用 3～4 列。
- H5 使用 2 列。
- 小于 420px 可使用 1 列。
- 金额卡片突出金额，说明文字保持简短。
- 负数使用红色，正向净值使用绿色。

## 7.2 查询行为

- 点击查询时，分页恢复第一页。
- 翻页只刷新列表，汇总可以复用同一查询结果或由接口同时返回。
- 重置筛选后同步刷新列表和统计。
- 请求失败时保留上一份成功数据或明确显示错误，不能变成假零值。

## 7.3 图表

- 分类和排名默认降序。
- 条目过多使用 `dataZoom`。
- Tooltip 展示完整值，不只展示格式化缩写。
- 图表 resize 兼容窗口变化和 H5 横竖屏切换。
- 空数据使用明确空状态，不初始化空白 ECharts。

## 7.4 Table

- 宽度保持 `100%`。
- 超出内容区时只在 Table 内部横向滚动。
- 继续保留列显示配置和自定义宽度。
- 新增列自动加入列设置，不能因为旧 localStorage 配置而永久不可见。

# 8. 测试计划

## 8.1 后端测试

每个新增汇总至少覆盖：

1. 无筛选统计。
2. 用户筛选。
3. 邮箱筛选。
4. 操作人筛选。
5. 时间范围筛选。
6. 多条件组合筛选。
7. 跨分页统计保持一致。
8. 空结果返回真实零值。

10. Asia/Shanghai 日期边界。

- 查询后列表和统计同步变化。
- 翻页不导致汇总只统计当前页。
- 重置恢复默认条件。
- H5 统计卡片不溢出。
- 图表 Tooltip、dataZoom 和 resize 正常。
- 表格仍支持列配置和横向滚动。
- 接口失败显示错误，不出现假零统计。

## 8.3 真实数据验收

至少选取一组可以人工核对的数据：

- 额度调整：同时包含调增和调减。
- 现金与赠送：同时包含纯现金、纯赠送和现金加赠送。

- 历史账：同时包含已关联、外部事件和审计孤儿。
- 对账：至少包含正常、告警和错误批次。

人工使用 SQL 或现有详情记录核对汇总结果，核对完成后记录验收样例。

实施前需要确认以下业务口径：

1. 异常中心是否需要增加正式的“已处理”状态及处理备注。
2. 经营账是否允许修改和删除，还是只允许新增和冲正。
3. Sub2API `actual_cost` 是用户实际消费还是平台实际成本。
4. 首页是否需要“收入消费差额”，以及是否需要纳入经营支出。
5. 赠送额度没有现金账关联时，是正常业务还是需要作为异常提示。
6. 对账 `warning` 是否允许视为业务可接受状态。
7. Sub2API 用户累计充值汇总是否需要展示，以及如何与本地实收入账区分。

在以上口径未确认前，可以先实施不依赖业务争议的 P0 项目：额度调整、现金账、赠送额度和经营账基础汇总。

# 11. 完成定义

每个页面只有同时满足以下条件才视为完成：

- 后端汇总基于完整查询结果。
- 列表和汇总筛选条件一致。
- 金额口径有明确名称和测试。
- 前端桌面端和 H5 均正常。
- Table 列配置、宽度和滚动不受影响。
- 接口失败不会展示假零值。
- 自动化测试通过。
- 使用线上真实数据完成至少一组人工核对。
